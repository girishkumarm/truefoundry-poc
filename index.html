How AI Coding Tools Actually Work

Every tool — Cursor, Cline, Aider, Claude Code — is just a smart UI making API calls

What the Developer Sees

Opens Cursor → selects code → types
"fix this bug" → code appears

What Actually Happens

1. Cursor collects your open files + cursor position 2. Builds a prompt: "Here's 3 files of context..." 3. HTTP POST → api.openai.com/chat/completions 4. Streams response back into your editor 5. That's it. No magic.

Why It Doesn't Matter for Coding

  • Typing the prompt takes 3-5 seconds — 200ms overhead is invisible
  • Streaming hides total time — developer sees tokens flowing continuously
  • Model thinking dominates latency (1-5s), not the gateway hop
  • Llama via Groq at 504ms is actually faster than most devs experience with direct Claude API (~1.2s)
Bottom line for the manager:
The ~200-400ms gateway overhead is imperceptible in a coding workflow. Developers will not notice any difference in Cursor or Aider. The model's own thinking time (1-5s) is what they feel, not the gateway.
Where it WOULD matter: Real-time chatbots with <200ms SLA. That's not the dev coding use case.

Claude Code CLI — The Gap

Claude Code requires Anthropic Messages API format — TrueFoundry only exposes OpenAI format

Why It Doesn't Work Directly

  • Claude Code uses Anthropic SDK internally (/v1/messages)
  • TrueFoundry only exposes OpenAI format (/chat/completions)
  • Claude Code supports ANTHROPIC_BASE_URL but needs Anthropic format at that URL
  • No --provider openai flag exists in Claude Code
Tested: TrueFoundry has no /anthropic/ endpoint. Tried 3 variations — all returned 404.

Workaround: LiteLLM Proxy

# Run LiteLLM as Anthropic→OpenAI translator pip install litellm # litellm_config.yaml model_list: - model_name: claude-sonnet-4 litellm_params: model: openai/claude-sonnet-4 api_base: https://tfy.internal.../openai api_key: <TF_KEY> # Start proxy litellm --config litellm_config.yaml --port 4000 # Point Claude Code to it export ANTHROPIC_BASE_URL=http://localhost:4000 export ANTHROPIC_API_KEY=sk-litellm claude

Status: Feasible but adds complexity. Could be deployed as a shared service for all devs.

Disadvantages

Trade-offs and limitations to consider

1. Gateway Latency Overhead

Every request goes through TrueFoundry's gateway, adding ~50-200ms. For real-time apps, this matters. For dev/testing, it's negligible.

2. Claude Code CLI Incompatible

Claude Code requires native Anthropic API. It cannot point to an OpenAI-compatible proxy. Developers using Claude Code still need direct Anthropic keys.

3. Feature Translation Gaps

Advanced features like extended thinking, prompt caching, and batch API may not be fully supported. OpenAI format is the common denominator.

4. Single Point of Failure

If TrueFoundry gateway goes down, all model access is lost. Mitigation: TF has HA setup, but this is an inherent gateway risk.

5. Model Naming Abstraction

Model names differ from provider names (e.g., claude-sonnet-4 not claude-sonnet-4-20250514). Virtual model routing may be opaque.

6. Debugging Complexity

When something breaks, you need to debug: your code → TF gateway → provider. Extra layer means extra troubleshooting.
